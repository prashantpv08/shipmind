# AGENTS.md

## Mission

Build the P0 hackathon MVP of **Axiom**, an AI Engineering Operating System that converts ambiguous business intent into structured requirements, architecture decisions, enterprise artifacts, controlled generated code, real verification evidence, and grounded Why / Why Not / Proof answers.

The authoritative product contract is `SRS.md`.

## Required Reading Order

1. `AGENTS.md`
2. `SRS.md`
3. `IMPLEMENTATION_BACKLOG.md`
4. `DEMO_SCENARIO.md`

## Scope Rules

- Implement P0 only until all P0 acceptance criteria pass.
- Do not create placeholder modules for P1 or P2 features.
- Do not add authentication, organizations, arbitrary repository ingestion, live meeting integrations, cloud provisioning, or multi-model routing.
- Implement one complete vertical journey using the NotifyFlow sample.
- Keep the application runnable after every milestone.

## Approved Stack

- TypeScript strict mode
- pnpm workspaces
- Next.js App Router
- Tailwind CSS
- Zod
- SQLite with Prisma or a documented equivalent
- Vitest
- Playwright
- Mermaid
- One model provider adapter

Do not change the stack without documenting a blocking issue and the smallest viable alternative.

## Architecture Rules

- Domain logic must not depend on React components or route handlers.
- Model provider code must be isolated behind an interface.
- All model responses must pass Zod validation before persistence.
- The canonical project graph is the source of truth.
- Generated Markdown documents are compiled views, not the primary database.
- Stable IDs must survive artifact regeneration.
- Truth-status transition logic must be centralized.
- Verification commands must be defined in code, never generated by the model.
- Generated files may be written only to allowlisted sandbox paths.
- The runner must enforce timeouts and bounded output.

## No-Hallucination Rules

- Never fabricate test results, coverage, scan findings, performance metrics, or command output.
- A model may explain tool output but may not modify measured values.
- Any unverified claim must use `AI_SUGGESTED` or `UNKNOWN`.
- Requirements grounded in source text must reference valid stored source spans.
- Do not invent source quotations.
- Failed commands produce `FAILED` evidence.

## UI Rules

- Every async action needs idle, loading, success, empty, and failure states.
- Do not show fake progress. Stage status must represent completed work.
- Do not communicate status by color alone.
- Traceability must have a non-graph fallback.
- Prioritize the hackathon story over dashboard density.

## Runner Rules

- No arbitrary shell commands.
- No package installation from model output.
- No arbitrary user code execution.
- Use a fixed notification-service template and fixed dependencies.
- Validate paths against a workspace root.
- Strip secrets from child-process environments.
- Kill timed-out processes.
- Store command, duration, exit code, parsed metrics, and bounded raw output.

## Expected Commands

P1 commands may include:

```bash

```

## Definition of Done

A task is complete only when:

1. It works through the visible user flow.
2. Input and outputs are validated.
3. Relevant tests pass.
4. Loading and failure states exist.
5. Traceability is created where required.
6. No fake evidence is displayed.
7. `pnpm lint`, `pnpm typecheck`, and relevant tests pass.
8. README or implementation notes are updated.

## Working Method

For each milestone:

1. State the goal and files to change.
2. Implement the smallest complete vertical slice.
3. Run the relevant commands.
4. Fix failures before expanding scope.
5. Summarize changed files, tests, known gaps, and the next milestone.

Do not silently reinterpret the SRS. Record open decisions in `docs/decisions/`.

## Engineering Discipline

These repository-local rules adapt the [Karpathy-inspired coding guidelines](https://github.com/multica-ai/andrej-karpathy-skills) and [Ponytail's lazy senior developer mode](https://github.com/DietrichGebert/ponytail). They complement, and never override, the Axiom product, safety, and verification rules above.

- Understand the requested behavior and trace its real callers before changing code. Surface material ambiguity and tradeoffs instead of silently assuming.
- Reuse an existing repository pattern, platform feature, standard-library capability, or installed dependency before adding an abstraction or dependency.
- Implement the smallest complete change that satisfies visible acceptance criteria. Avoid speculative flexibility, unrelated cleanup, and drive-by refactors.
- Fix shared root causes at the narrowest correct boundary rather than patching one symptom or duplicating guards across callers.
- Preserve validation, security, accessibility, evidence integrity, error handling, and data-loss protection; brevity never excuses weakening them.
- Make every non-trivial change leave the smallest useful runnable regression check. Define success criteria up front and loop until they pass.
- Every changed line must trace to the current goal. Remove only dead code made obsolete by the current change; report unrelated debt without editing it.
